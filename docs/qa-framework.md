# CortaLoom Quality Assurance Framework

## The Problem: The Cost of a False Positive
In prior authorization, the cost of an AI hallucination or a missed policy nuance is severe: a denied claim, delayed patient care, and lost trust from the ASC billing coordinator. If CortaLoom generates a "Green" Payer Readiness Score for a packet that is subsequently denied because the MRI was 7 months old instead of 6, the product's core value proposition is broken.

Quality cannot be a retrospective metric. It must be built into the product's architecture as a continuous feedback loop.

## The Four Pillars of CortaLoom Quality

### 1. Source Grounding (The "Show Your Work" Principle)
The AI must never output a fact (a date, a diagnosis, an imaging result) without pointing directly to the sentence in the medical record where it found it.
- **Implementation:** Every extracted data point in the Surgical Case Card must be clickable. When clicked, the UI must highlight the exact source sentence in the original PDF viewer.
- **Why it matters:** It shifts the coordinator's job from "trusting the AI" to "verifying the AI." It takes 2 seconds to verify a highlighted sentence; it takes 10 minutes to hunt for it manually.

### 2. Human-in-the-Loop Override (The Feedback Engine)
When the AI gets something wrong, the coordinator must be able to fix it immediately within the UI, and that fix must be captured as training data.
- **Implementation:** Every field in the extracted payload (e.g., "Physical Therapy Dates," "Conservative Treatment Duration") must be editable by the user. 
- **The Data Moat:** When a user overrides a value, the system must log the original AI prediction, the user's correction, and the source document context. This creates a proprietary dataset of edge cases that can be used to fine-tune future models or improve prompt templates.

### 3. Closed-Loop Outcome Tracking (The Ultimate Truth)
The only metric that truly matters is the final prior authorization outcome from the payer (Approved, Denied, Pending Info).
- **Implementation:** The system must track the lifecycle of every generated packet. After a packet is marked "Submitted," the coordinator must eventually log the outcome.
- **The Metric:** We track the **First-Pass Approval Rate** of CortaLoom-generated packets vs. the ASC's historical baseline. If a packet generated by CortaLoom is denied, the system must prompt the user for the denial reason (e.g., "Missing conservative treatment," "Imaging too old").
- **The Feedback Loop:** Denials of "Green" packets are P0 system failures. They indicate a gap in our Payer Policy Database that must be patched immediately.

### 4. Automated Regression Testing (The Safety Net)
As we update prompts, switch LLM providers, or add new payer policies, we must ensure we don't break existing extraction logic.
- **Implementation:** A suite of synthetic, anonymized "Golden Patient Records" (like the ones generated in `synthetic-demo-data.json`) covering every edge case (missing PT, old imaging, conflicting diagnoses).
- **The Process:** Every time the extraction prompt or schema is updated, the system must automatically run the Golden Records through the new engine and compare the output against the known correct answers. If accuracy drops, the build fails.

## The Immediate Action Items (GitHub Issues)

To implement this framework, three specific features need to be built into the current product:

1. **Source Highlighting UI:** The frontend must support clicking an extracted value and scrolling/highlighting the source PDF.
2. **User Override Logging:** The backend must capture and store any edits made by the user to the extracted payload.
3. **Outcome Tracker:** A simple UI flow to log the final Payer decision (Approved/Denied) and the denial reason, feeding back into the analytics dashboard.

By implementing these three features, CortaLoom transitions from a static extraction tool to a self-improving system where every user interaction makes the core engine smarter.
